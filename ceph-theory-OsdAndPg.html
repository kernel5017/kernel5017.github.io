ceph-theory-OsdAndPg

Created2020-12-17|Updated2020-12-21|ceph
|Post View:

OSD status

一个OSD的status有以下几种：

  • in: in the cluster
  • out: out of the cluster
  • up: up and running
  • down: not running
上一个状态 当前状态 行为动作
in out ceph will migrate PGs to other OSDs
in/out out ceph will not assign PGs to this OSD

ceph is NOT Health_OK

如果你的ceph集群不是Health_OK状态，那么可能有以下几种情况：

  • You haven't started the cluster yet
  • The OSDs are in the peering when you just started/restart the cluster
  • You just added/removed an OSD
  • You just have modified your cluster map

检查osd状态常用命令

  • ceph osd stat
  • ceph osd tree
  • ceph osd df

PG Sets

  • If a POOL requires 3 replicas of a PG, CRUSH may assign them to osd.1, osd.2, osd.3

  • CRUSH seeks a pseudo-random placement that will take into account failure domains you set in CRUSH MAP

  • Acting Set

    • As a set of OSDs that contain the replicas of a PG

When a OSD in Acting Set is down

Action Ceph Behavior Note
When You added or removed an OSD Ceph reassigned the PGs to other OSDs==> Changing the composition of the Acting Set==> Spawning the migration of data with “backfill” process
When OSD was down, was restarted Ceph is now recovering
An OSD in the Acting Set is down or unable to service requests Another OSD has temporarily assumed it duties

Up Set

Up set: the set of OSDs that will actually handle the requests。

Up Set和Acting Set世纪上是相同的。

When Ceph Behavior
Up Set != Acting Set 1.Ceph is migrating Data
2.An OSD is recovering
3.The cluster is rebalancing itself
4.Problem: HEALTHY WARN with “stuck state”

Check PG Status

PG ID的格式为：{pool-num}.{pg-id}

比如6.39===> pool 6, id 39

  • 列出所有PG

  • 查看PG对应的acting set或up set

    pg2.d对应的acting set和up set为osd.0,osd.2,osd.3

要点

peering

  • 写入data之前，PG的状态必须是active+clean，所以必须让这个PG内的所有OSD Daemons都同意目前的状态。这个验证同意的工作在ceph的设计里称之为peering
    • 做法：就是让PG中Acting set中的第一个OSD(Primary OSD)向second OSD,third OSD进行验证active+clean状态
  • peer完毕后，OSDs也会将状态汇报给Monitor

PG Status

Ceph does NOT HEALTH_OK

When Note
[Pool] Just create a pool and PG haven’t peered yet
[Recovering] The PG are recovering
[OSD] Just added an OSD or removed an OSD
[CRUSHMap] Just modified the CRUSH map and your PG are migrating
[Status error] There is inconsistent data in different replicas of a PG
Ceph is scrubbing a PG’s replicas

Query PG ceph pg {poolnum}.{pg-id} query

Creating PG

状态

peering

When Ceph is Peering a placement group, Ceph is bring the OSDs that store the replicas of the placement group into agreement about the state of the object and metadata in the placement group

Activing

  • When a PG is in the active state.

    • The data in the PG is generally available in the primary placement group and the replicas for read and write operations

当一个PG处于active状态，表示这个PG的primary placement group已经处于可用状态，且replicas做好读写准备

Clean

  • When a PG is in the clean state,

    • The primary OSD and the replica OSDs have successfully peered
    • There are no stray replicas for the PG //满足replica指定的值，所以pg都映射到了指定数量的OSD
    • Ceph replicated all objects in the PG the correct number of times //PG中的所有objects都被复制了正确的数量

Degraded

A primary OSD writes the object to storage, the PG will remain in a degraded state until the primary OSD has received an acknowledgement from the replica OSDs that Ceph created the replica objects successfully。

如果PG中的object没有被复制够规定的数量，就会出现这个状态。比如replicas=3,但是pg只有两份

Active+ degraded

  • An OSD may be active even though it doesn’t hold all of the objects yet
  • If an OSD goes down, Ceph marks each PG assigned to the OSD as degraded

Recovering

  • When an OSD goes down, its contents may fail behind the current state of other replicas in the PG
  • Ceph was designed for fault-tolerance at a scale where hardware and software problems are ongoing

表示objects正在replica之间进行同步/迁移

Backing Filling

  • When a new OSD joins the cluster, CRUSH will assign PG from OSDs in the cluster to the newly added OSD
State Behavior
backfill_wait a backfill operation is pending
backfill a backfill operation is underway
backfill_too_full a backfill operation was requested, but couldn’t be completed due to insufficient storage capacity
Setup Behavior Default Value
osd max backfills the maximum number of concurrent backfills to or from an OSD 10
osd backfill full ratio Enables an OSD to refuse a backfill request if the OSD is approaching its its full ratio (85%, by default). 85%

Remapped

When the Acting Set that services a PG changes, the data migrates from the old acting set to the new acting set。

the PG is temporarily mapped to a different set of OSDs from what CRUSH specified

Stale

  • The ceph-osd daemons may also get into a stuck state where they aren’t reporting statistics in a timely manner (e.g., a temporary network fault) while Ceph uses heartbeat to ensure the system are running

    • If the primary OSD of a placement group’s acting set fails to report to the monitor or
    • If other OSDs have reported the primary OSDs down

==> the monitor will mark the placement group stale

the PG is in an unknown state - the monitors have not received an update for it since the PG mapping changed.

Identifying Troubled PGs

State Behavior
Unclean Placement groups contain objects that are not replicated the desired number of times. They should be recovering
InActive Placement groups cannot process reads or writes because they are waiting for an OSD with the most up-to-date data to come back up.
Stale Placement groups are in an unknown state, because the OSDs that host them have not reported to the monitor cluster in a while (configured by mon osd report timeout).

ceph pg dump_stuck [unclean | inactive | stale]
