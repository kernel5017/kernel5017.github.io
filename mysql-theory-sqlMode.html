SQL mode可以设置global和session级别，所以每个client可以单独设置自己的SQL mode。

SQL mode会影响MySQL支持的SQL语法和执行的数据验证检查

设置SQL mode

​ 可以在数据库启动时添加选项来设置sql mode，比如–sql-mode=”modes” ；也可以在配置文件my.cnf中设置sql-mode=”modes”,设置多个值时候使用逗号分割。如果不想设置任何sql mode,那么使用–sql-mode=””,或者配置文件中写sql-mode=””。

​ 在数据库运行期间改变sql mode可以通过set语句来修改：

​ 当设置global级别的sql mode时，需要SUPER权限，设置后会影响所有客户端，但是只影响设置后新连接进来的客户端，不影响老的客户端。设置session级别的sql mode，只影响当前client,每个client都能随时改变自己的sql_mode。

查看当前global或session级别sql_mode:

警告！

在创建了分区表并插入数据之后，如果改变sql_mode可能会导致分区变发生重大改变，并且会丢失或者损坏数据。强烈建议，当创建分区表后不要再更改sql_mode。

当主从复制时有分区表的情况下，不同的sql mode也会导致问题。所以如果使用的分区表，那么主从的sql_mode必须设置为一致的。

重要sql mode介绍

当提到strict mode时，我们认为STRICT_TRANS_TABLES或STRICT_ALL_TABLES 至少启用了一个，或者都启用了。

TRADITIONAL

​ 此模式使MySQL更像一个“传统”的SQL数据库。简单描述就是，在这种sql mode下，如果insert了一个非法的值，那么会返回一个error而不是warning。

全部sql mode介绍

ALLOW_INVALID_DATES

​ 不会对日期进行完整检查，只会检查month是否在1-12范围，检查day是否在1-31范围内。比如我插入一条’2020-02-30’这条数据，因为day的检查范围是1-30，所以设置了这个模式后，这条数据是可以插入的，但是实际是不合法的。

​ 这个模式只对类型为date和datetime类生效，不会对timestamp列生效，因为timestamp列总是需要一个合法的日期。

​ 当关闭strict_mode后，如果插入’2004-04-31’（不存在这一日期），那么会将它改变为’0000-00-00’进行插入，并产生告警；如果开启了strict mode，那么插入会产生一个error;如果想允许插入这个非法日期，那么开启allow_invalid_dates

ANSI_QUOTES

​ 一般情况下，我们插入字符串时候会使用双引号包围字符串，比如insert into t select “abc”;这样插入的就是字符串abc。但是如果启用这个sql mode后，双引号就不能作为字符串包围使用，它是作为字符串本身来使用，启用之后如果想插入字符串abc要这样插入：insert into t select ‘abc’;如果你使用双引号”abc”则会报错，它认为”abc”是一个字符串整体，所以你需要使用单引号把这个整体包围：insert into t select ’”abc”‘;这样插入的结果是”abc”，如下所示：

ERROR_FOR_DIVISION_BY_ZERO

这个sql mode影响的是除以0的结果，对于insert,update中包含除以0的操作来说，还取决于是否开启了strict sql模式。

  • 如果error_for_dIvision_by_zero没有开启，那么一个数除以0会插入NULL,且没有告警
  • 如果开启error_for_division_by_zero,那么一个数除以0会插入NULL,且返回warnings
  • 如果开启了error_for_division_by_zero和strict mode,除以0会报error，无法插入数据；除非使用insert ignore和update ignore,当使用ignore后，会报一个warning，然后插入NULL

对于select中包含除以0时会返回NULL。开启ERROR_FOR_DIVISION_BY_ZERO只会使select产生一个warning,且不管是否是strict mode都是如此。

HIGH_NOT_PRECEDENCE

not操作符的优先级对于表达式not a between b and c来说，被解析为not (a between b and c)；对于老版本mysql，这个表达式被解析为(not a) between b and c;如果想使用老版本的这种优先级策略，那么可以开启high_not_precedence;

IGNORE_SPACE

允许函数名和”(“左括号之间有空格。这将导致built-in函数名会被作为保留字来处理。因此，如果开启了这个sql mode，那么与函数名相同的标识必须用反引号括起来。举例

也就是说，如果保留字和( 之间保留一个空格，是可以创建的。

总结

  • 如果不开启ignore_space,使用保留字创建表名，可以在表名和( 之间添加一个空格，即可创建成功
  • 如果开启ignore_space,使用保留字创建表名，必须使用反引号包围表名
  • ignore_space只是针对built-in function，不会对user-defined function或者stored function起作用。

NO_AUTO_CREATE_USER

如果grant 语句不使用identified by 或identified with语句设置密码，那么grant语句不能自动创建新用户

NO_AUTO_VALUE_ON_ZERO

如果不开启no_auto_value_on_zero，当向一个自增列插入0或NULL时，都会引起自增列的增长，比如：

当开启no_auto_value_on_zero时，插入0不会引起自增列的增长，而是插入数值0

总结

  • 如果自增列需要插入数字0，那么需要设置此sql mode
  • 当使用mysqldump一个自增列的表，然后重新导入时，如果遇到0会进行自增，而不是插入0，可能导致和原表数据不一致，所以需要在倒入钱开启这个sql_mode。现在mysqldump会自动添加这个语句到备份文件中，在倒入时会自动执行开启，所以就不用管了。

NO_BACKSLASH_ESCAPES

当开启no_backslash_escapes后，反斜线’'不在作为转义符使用，而是作为普通的字符串来使用。

NO_DIR_IN_CREATE

启用no_dir_in_create,在创建表的时候，如果使用了index directory和data directory参数，这些参数会被忽略。也就是不会在指定目录生成文件和索引，还是使用系统默认的datadir

NO_ENGINE_SUBSTITUTION

控制create table或alter table时候的默认存储引擎，如果

  • 当关闭no_engine_substitution时候，如果使用create table命令，那么会使用默认的存储引擎创建表，默认存储引擎由default_storage_engine参数指定，如果默认存储引擎不可用，则产生一个warning，但是表会创建成功；对于alter table语句，会产生一个warning，并且不会执行alter动作。
  • 当开启no_engine_substitution时候，如果默认存储引擎不可用，则create table或alter都不会被成功执行。

NO_UNSIGNED_SUBTRACTION

两个整数之间的减法，如果其中一个整数为unsigned(无符号)的，那么默认会产生一个无符号的结果，如果结果是一个负数(负数是有符号的)，那么会报错，比如：

如果开启了NO_UNSIGNED_SUBTRACTION，那么结果是负数，不报错，比如：

就记住，对于unsigned值的操作，结果都是unsigned的。如果想要结果不是unsigned的，就开启这个sql mode.

NO_ZERO_DATE

no_zero_date控制着’0000-00-00’是否为合法的日期，不过还依赖于sql mode是否为strict模式，这两个结合起来控制着这个行为。

  • 如果没有开启no_zero_date，那么插入’0000-00-00’是允许的，并且不会产生告警
  • 如果开启了no_zero_date,那么插入’0000-00-00’是允许的，并且产生告警
  • 如果开启了no_zero_date，并且开启了strict mode,那么‘0000-00-00’是不允许插入的，插入会返回一个error,除非你添加了ignore关键字。如果使用了ignore关键字，那么insert ignore或者update ignore ‘0000-00-00’时可以插入，但是产生一个warning.

从5.7.4开始，no_zero_date已经被弃用。mysql5.7.4到5.7.7之间，no_zero_date如果启用，不会有任何作用。从mysql5.7.8和之后开始，no_zero_date和5.7.4之前的作用一样，也就是上面介绍的几点功能一样。它应该与strict mode结合起来使用。

NO_ZERO_IN_DATE

no_zero_in_date控制是否允许year部分为非0，但是month或者day为0.比如’2010-00-01’,’2010-01-00’但是不是’0000-00-00’这种。这个模式也是结合strict sql使用。

  • 如果未开启，那么允许为0，不会有warning
  • 如果开启，产生告警
  • 如果开启，且开启strict mode，不允许插入，并返回error。使用ignore可以避免发生error

ONLY_FULL_GROUP_BY(p932)

从5.7.5开始，默认的sql mode包含only_full_group_by。在这之前是是默认不包含的。

PAD_CHAR_TO_FULL_LENGTH

默认情况下，当查询数据时，对于char类型来说，末尾的空格是被去掉的，如果PAD_CHAR_TO_FULL_LENGTH 被开启了，那么末尾的空格不会被去掉，并且会填满整个长度。整个模式对于varchar来说无效。

PIPES_AS_CONCAT

将||作为字符串连接符，和concat()一样，而不是OR的同义词

REAL_AS_FLOAT

Treat REAL as a synonym for FLOAT. By default, MySQL treats REAL as a synonym for DOUBLE.

STRICT_ALL_TABLES

Enable strict SQL mode for all storage engines. Invalid data values are rejected.

STRICT_TRANS_TABLES

Enable strict SQL mode for transactional storage engines, and when possible for nontransactional
storage engines.
