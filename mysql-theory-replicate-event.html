mysql主从复制-event格式

Created2020-12-16|Updated2020-12-16|mysql
|Post View:

Binary log概述

每个mysql的binary log都是由开头的4字节的魔术数和一个一个的event组成的。在linux中可以查看binary log的魔术数：

”fe 62 69 6e“的16进制编码就是binary log的魔术数，这就是就每个binary log总是由# at 4开头的原因。

所有的event都由相同的结构组成，包括event header,event data,event footer:

mysql的event有多个版本，老版本我们这里就不讨论了，我们只看目前使用的版本，mysql5.0和更高的版本使用的是V4版本。下面是v4版本的event结构：

event header 和event footer

所有event的header和footer的大小、结构都是固定不变的，他们的含义如下：

  • timestamp:**固定4字节，是新纪元时间（1970年1月1日0时0分0秒）以来的秒数。这个时间是命令发起的时间**。什么是命令发起的时间呢？就是执行计划生成后，开始实际执行语句的时间。对于语法检查，权限检查，优化器生成执行计划的时间都包括在里面了。

    如果手动开启一个事务，那么这个时间记录的不是执行begin时的时间，而是真正执行命令的时间。举例，假如我手动开启一个事务，插入两条数据，然后commit，那么对应的二进制日志如下：

  • type_code固定1字节，event有很多类型，这个就是不同类型event的编码，每种event都有自己的编码。

  • server_id固定4字节，代表了生成这个event的mysql的server id。如果从库开启了log_slave_updates参数，从库将主库发过来的event写入到binary log，这个event的server id也是主库的server id。这样可以避免主从无限循环。比如三个mysql server，M1,M2,M3,他们的server id分别为1，2，3.主从关系如图

    当客户端对M1发起一个insert语句后，产生了一个event,这个event中的server id=1。然后这个event发送到M2，M2重放后写入到自己的binary log，写入的server id也是1;然后M2将event发送到M3,同样M3写入到日志时候server id也是1。最后M3将event又发送到M1，这时M1发现这个产生这个event的server id为自己，所以他就不会执行这个event。所以避免了无限循环。

  • event_lenth固定4字节，整个event的长度

  • next_position固定4字节，下一个event在master的binary log中的位置

    对于v4版本binlog来说： 是指从binlog文件开始计算到事件结束的偏移量。所以binlog中的第一个event的next_position=4+event_length,对于event n和event n+1来说，next_position(n+1)=next_position(n)+event_length(n+1)。

    对于v4版本relay log来说：是指从binlog文件开始计算到时间结束的偏移量，和在master的binlog中一样。

    ​ slave的relay log可能和master的binlog不一样，所以relay log中的next_postion可能和master中的binlog是不一样的。如果event n和event n+1都是源于master，那么next_position(n+1)=next_position(n)+event_length(n+1)。

    ​ next_position在slave上有两个用途：

    ​ 1.for SHOW SLAVE STATUS to be able to show coordinates of the last executed event in the master’s coordinate system.

    ​ 2.for START SLAVE UNTIL MASTER_LOG_FILE=x, MASTER_LOG_POS=y, so that the master’s coordinates can be used.

    对于5.0或者更高版本，next_position在mysqlbinlog和show binlog events中叫做“end_log_pos”

  • flags固定2字节，某些event包含这个标志，比如format_description_log_event中，log_event_binlog_in_use_f表示当前binary log是当前写入文件

  • event footer中crc:固定4字节。这部分是整个event的一个crc校验码，用于表示event的完整性。

详细解析

使用mysqlbinlog –hexdump解析一下binlog日志：

  • timestamp“a1 a7 d9 5f”小端显示，实际为16进制的“5fd9a7a1”,转为10进制等于”1608099745”,使用date命令看一下结果：

  • type1e转换为10进制为30

  • master id实际为16进制的“00000064”，转换为10进制为100

  • size十六进制为00000028,转换为10进制就是40字节，也就是本event的长度，下一个event开始为378，减去本event开始338，正好为40

  • master pos转为10进制为378，就是下一个event开始的位置。

event type code

query_event=2

在binlog_format=statement模式下，query_event记录实际执行的语句。在binlog_format=row模式下，不记录任何语句相关的信息，但是DDL始终是记录的语句。

format_description_event=15

说明了binary log的版本信息，总是包含在每一个binary log的开头位置

xid event=16

当事务提交的时候记录这个eventqizh9ong携带的XID信息

table_map_event=19

包含了table_id和表名的映射，比如：

#201216 10:45:19 server id 100 end_log_pos 341 CRC32 0x95e54e81 Table_map: test.test mapped to number 139

write_rows_event=30

insert语句生成的event,包含插入的实际数据，row模式才有

update_rows_event=31

update语句生成的evetn,包含update之前和之后的数据，row模式才有

delete_rows_event=32

delete语句生成的event，包含实际需要删除的数据，rwo模式才有

gtid_log_event=33

如果开启了GTID模式，那么会生成关于gtid的信息，并且携带last commit和seq number作为MTS并行回放的依据

anonymout_gtid_log_event=34

在未开启GTID模式的时候生成，并且携带last commit和seq number作为MTS并行回放的依据

previous_gtids_log_event=35

说明前面所有的binary log包含的gtid set,relay log则代表IO线程收到的GTID SET。
