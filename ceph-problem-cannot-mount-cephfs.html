无法mount cephfs

情况描述

​ 测试环境有4个ceph节点、5个osd，数据写入大概13G左右，集群整体还有5个多G可用空间。由于测试其他问题，需要删除一个ceph node节点，恢复为3个osd。于是将挂载目录中的数据全部删除，接着停止osd，剔除osd。

​ 恢复到3个节点，3个osd后，集群健康状态为OK。但是写入数据出现问题，刚开始使用dd命令创建一个1G的文件，生成到大概300M左右，就卡住不动了。然后创建了几个小文件，写入到十几K的时候也不动了。于是umount这个目录，发现无法卸载，最后重启后重新挂载，发现无法挂载了，报错是connection time out。

​ 查看ceph日志，没有什么特别的异常。于是查看client机器的日志，发现报错：

原因

这个报错是因为挂载cephfs的客户端缺少对flag 400000000000000的支持。

ceph tunables

​ 我们知道ceph client通过crush算法来寻找自己需要的数据存储在哪个osd上。随着时间的推移，ceph crush算法有很多改进，每个ceph版本都有相应的crush算法。为了能够让的不同版本支持不同的crush算法，ceph引入了tunable options,这些选项控制着使用哪个版本的crush算法。

​ 为了使用更新版本的tunables,clients和servers都必须支持新版的CRUSH。基于这个原因，引入了以ceph版本命名的profiles文件。比如firefly tunables是Firefly版本支持的，它不能在老版本上工作，比如Dumpling（比Firefly早）。

​ 一旦给定的可调参数集改变了默认的legacy，那么ceph-mon和ceph-osd将阻止不支持新的CRUSH特性的老客户端连接到集群。

具体看文档https://docs.ceph.com/en/latest/rados/operations/crush-map/

解决

设置feature flags不需要flag 400000000000000的支持

关于Feature Set Mismatch Error on Ceph Kernel Client问题总结

根据你的kernel版本不同，可能会出现你的kernel无法提供ceph集群需要的一些特性。或者或者你的ceph client需要的一些特性，但是ceph cluster无法满足，所以会出现”feature set mismatch”这个报错。

解决：Upgrade kernel client up to 3.9 or unset hashpspool : ceph osd pool set rbd hashpspool false
